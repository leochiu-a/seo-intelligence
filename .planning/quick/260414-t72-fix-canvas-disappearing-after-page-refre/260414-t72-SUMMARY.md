# Quick Task 260414-t72: Fix canvas disappearing after page refresh — Summary

**One-liner:** Replaced isRestoring ref with isFirstRender guard so save effect skips only its first invocation, not the re-render triggered by setNodes, preventing localStorage from being overwritten with empty state in StrictMode.

## What Was Built

### Root Cause

Both `useEffect` hooks (restore and save) run synchronously within the same React commit on mount. The restore effect:
1. Calls `setNodes(restoredNodes)` — schedules a state update (async)
2. Sets `isRestoring.current = false` — immediately

The save effect then runs in the same commit:
- `isRestoring.current` is `false` → skip guard not triggered
- `nodes` is still `[]` → saves `{nodes: [], edges: []}` to localStorage

In StrictMode (production and dev), effects are invoked twice. The second invocation of the restore effect reads the now-empty localStorage and restores nothing — canvas disappears.

### Fix

Replaced `isRestoring` with `isFirstRender`:

```ts
// Before (buggy):
const isRestoring = useRef(true);
// ...restore effect sets isRestoring.current = false BEFORE save effect reads nodes
useEffect(() => {
  if (isRestoring.current) return;
  localStorage.setItem(STORAGE_KEY, ...);
}, [nodes, edges]);

// After (fixed):
const isFirstRender = useRef(true);
// ...restore effect no longer touches the flag
useEffect(() => {
  if (isFirstRender.current) {
    isFirstRender.current = false;
    return;
  }
  localStorage.setItem(STORAGE_KEY, ...);
}, [nodes, edges]);
```

`isFirstRender` unconditionally skips the first invocation, regardless of restore timing. All subsequent re-renders (including the one triggered by `setNodes`) save the correct state.

### Test Strategy

The regression test renders `<App />` inside `React.StrictMode` to reproduce the double-invocation scenario. It:
1. Seeds localStorage with 2 nodes
2. Mounts App (StrictMode double-invokes effects)
3. Asserts localStorage still contains 2 nodes (not overwritten to `[]`)
