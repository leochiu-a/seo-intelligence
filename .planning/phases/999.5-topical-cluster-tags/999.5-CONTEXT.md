# Phase 999.5: Topical Cluster Tags + Same-Cluster Edge Weighting - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 999.5 adds three layered capabilities:

1. **Cluster/tag data model** — Each node carries an optional `tags: string[]` field. PMs assign tags via the existing EditPopover using a free-text + autocomplete input (same pattern as Phase 6 placement autocomplete).
2. **Same-cluster PageRank bonus** — Inside `calculatePageRank`, an edge whose source and target share **at least one tag** contributes with `linkCount × CLUSTER_BONUS_FACTOR` (1.5×). Non-overlapping / unassigned stays at raw `linkCount`. Implements the topical-authority modeling gap called out in the 999.5 backlog note.
3. **Visual grouping** — Left-edge color stripe on each node card (deterministic palette per tag, split into bands for multi-tag), same-cluster edges colored in muted cluster tone, sidebar prefix dots, and a new "By cluster" section in the filter panel that AND-combines with the existing placement filter.

**In scope:** Multi-tag data model (tags array), free-text + autocomplete entry, global-node participation, multiplicative bonus via `CLUSTER_BONUS_FACTOR`, left-edge stripe (split bands), same-cluster edge coloring, sidebar dots, filter-panel "By cluster" section, JSON export `tags` field, cluster-name→palette hash.

**Out of scope / deferred:** URL-prefix auto-inference (→ Phase 999.11), embedding-based semantic similarity (→ Phase 999.12), anchor-text typing (→ Phase 999.6), user-tunable bonus factor, per-cluster rename/merge UI, cluster-based CSV export columns.

</domain>

<decisions>
## Implementation Decisions

### Cluster Data Model
- **D-01:** Node carries `tags: string[]` (multi-tag). Storage is an array of cluster name strings; empty/missing array means "unassigned."
- **D-02:** Tag entry UX is **free-text + autocomplete**, mirroring Phase 6 placement autocomplete exactly. Implement a parallel `collectClusterSuggestions(nodes, currentNodeId): string[]` utility in `graph-utils.ts` and wire through `EditPopover` with a Base UI `Autocomplete`.
- **D-03:** Cluster is **optional** on every node. No validation requiring at least one tag.
- **D-04:** Global nodes **participate fully** in clusters. They can carry tags; the synthetic global-injection edges get the same-cluster bonus when the non-global source's tags intersect the global node's tags.
- **D-05:** Same-cluster test = **set intersection ≥ 1**. If `source.tags ∩ target.tags` is non-empty, apply the bonus **once** (bonus does not scale with overlap size).

### Weight Math
- **D-06:** Bonus is **multiplicative on effective linkCount** inside `calculatePageRank`:
  - Same-cluster edge → `effectiveLinkCount = edge.linkCount × CLUSTER_BONUS_FACTOR`
  - Cross-cluster / unassigned edge → `effectiveLinkCount = edge.linkCount`
  - Apply inside both the `inbound` traversal and `totalWeightedOut` precompute so rank transfer stays math-consistent.
  - Applies identically to the synthetic global-injection edges.
- **D-07:** `CLUSTER_BONUS_FACTOR = 1.5`.
- **D-08:** Bonus exposed as a **module-private const** in `graph-utils.ts`, colocated with `DAMPING`, `MAX_ITER`, `EPSILON`, `OUTBOUND_WARNING_THRESHOLD`:
  ```ts
  const CLUSTER_BONUS_FACTOR = 1.5;
  ```
  No UI knob. Matches Phase 8/10 precedent.
- **D-09:** Phase 10's outbound-link warning keeps using **raw `linkCount`**. Cluster bonus is a PageRank modeling trick — it does not represent physical links on the page, so the >150 threshold stays semantically "real links."

### Visual Treatment
- **D-10:** Cluster color is **auto-assigned via deterministic hash** of the tag name → index into a fixed palette. Same tag name always maps to the same color across scenarios / reloads. PM never picks colors manually.
- **D-11:** Node card shows:
  - **Left-edge color stripe** (4px) — orthogonal to Phase 2 scoreTier border; does not conflict.
  - **Tag chips in the subtitle row** — one compact chip per tag (e.g. `food · taipei`), integrated into the existing subtitle indicator flex.
- **D-12:** Edge coloring:
  - Same-cluster edges → muted cluster color (use source-side primary tag, or any shared-tag color — Claude's discretion, pick a deterministic rule).
  - Cross-cluster / unassigned edges → existing default gray. **No penalty framing**, no dimming.
- **D-13:** Cluster stripe **persists at full saturation** when the node is dimmed by the placement/cluster filter. Stripe is a structural marker; the filter dims the card body only.
- **D-14:** Multi-tag stripe is **split into vertical bands**, one band per tag. Cap at **3 visible bands**; 4+ tags render first 3 + a `+N` indicator (rare).

### Sidebar & Filter Integration
- **D-15:** Unassigned nodes (`tags` empty/missing) form **no implicit bucket**. Their set intersection with anything is empty → no bonus ever applies. Zero incentive to skip tagging is preserved; no UX reward for sloppy data.
- **D-16:** Score sidebar shows **up to 3 small colored dots** before the URL template on each row (matching the stripe palette). No text chips in the sidebar — dots only, to keep row height unchanged.
- **D-17:** JSON export includes `tags: string[]` per node; import must accept and round-trip it (with backward-compat: missing `tags` = empty array). CSV export stays unchanged — Phase 3 `url_template, page_count, score` columns are frozen.
- **D-18:** FilterPanel extends with a **new "By cluster" section** in this phase. Lists unique tag names (deduplicated across all nodes) as checkboxes, mirroring Phase 7's placement grouping.
- **D-19:** When both placement and cluster filter dimensions have active selections, they **AND-combine**: a node stays at full opacity only if it matches at least one active placement AND at least one active cluster. Within a single dimension, multiple selections OR together (standard).

### Claude's Discretion
- Exact function signatures — suggested: `hasSameCluster(sourceTags, targetTags): boolean`, `collectClusterSuggestions(nodes, currentNodeId): string[]`, `getClusterColor(tag): string` (hash-based palette lookup).
- Exact palette — 8–10 visually distinct colors, tested for accessibility against card backgrounds and the existing scoreTier palette. Avoid colors already heavily used (red = over-linked, amber = weak, indigo = neutral tier).
- Stripe-band sizing when 2–3 bands render (equal heights? top-weighted?).
- Tag chip styling (size, padding) to keep subtitle row height parity with Phase 8/10 indicators.
- Edge coloring tie-break when source and target share 2+ tags (pick source-side primary, or first shared tag alphabetically — just be deterministic).
- TDD cases for `calculatePageRank` cluster bonus: (1) two same-cluster nodes → bonus applied; (2) two different-cluster → no bonus; (3) one untagged, one tagged → no bonus; (4) both untagged → no bonus; (5) multi-tag overlap on ≥1 tag → bonus applied once, not multiplied; (6) global source with matching cluster on target → synthetic edge bonused; (7) global source with non-matching cluster → synthetic edge unbonused.
- Whether to add `tags` to `UrlNodeExtendedData` or keep it in base `UrlNodeData` (likely base, since it persists).
- localStorage migration: no migration needed — `tags` defaults to `[]` when absent on scenario load.

</decisions>

<specifics>
## Specific Ideas

- **Palette seed:** Keep the scoreTier colors (green/amber/red/indigo) out of the cluster palette to avoid false "this node is mid-tier"-style misreads. Candidate hues: teal, violet, pink, sky, lime, orange, rose, cyan. Aim for ~8 distinct, desaturated tones that read "tag" not "status."
- **Hash → palette:** Simple DJB2 or FNV-1a hash of the cluster name mod palette length. Avoid non-deterministic hashing (e.g., JS `Math.random`). Snapshot-testable.
- **Multi-tag stripe bands:** Equal-height vertical division of the 4px stripe. 1 tag = full stripe. 2 tags = two stacked halves. 3 tags = three equal thirds. 4+ = 3 bands + tiny `+N` label inline in subtitle (no band for overflow).
- **Edge color rule (deterministic):** When source and target share multiple tags, the edge takes the color of the **first shared tag in sorted order**. Pure function, easy to test.
- **Autocomplete polish:** Suggest tags only when OTHER nodes have tags defined (mirror Phase 6 PLACE-04 empty-state rule).
- **Filter panel empty state:** "By cluster" section hidden entirely when no node in any scenario has tags — reveals itself after first tag is added (matches placement autocomplete's discoverability spirit).
- **Export backward compat:** JSON import must default missing `tags` to `[]`. No localStorage migration needed.

</specifics>

- **Scenario-level cluster palette customization** — PM picks palette per scenario. Out of scope; deterministic hash is shipped default.
