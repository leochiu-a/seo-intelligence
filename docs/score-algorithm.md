---

## 整體流程

每次你新增節點、拉連線、或改任何屬性，系統會立刻（不需要按按鈕）依序做以下事情：

1. **算每個節點總共放出幾條連結**
2. **跑 PageRank，算出每個節點的重要性分數**
3. **把分數換成 high / mid / low 三個等級**
4. **找出分數異常低的節點（Weak）**
5. **從首頁出發，BFS 算每個節點要幾步才能到達**
6. **找出完全沒有人連它的孤島節點**
7. **對每個節點評估三項健康指標：Links（外連數）、Depth（爬取深度）、Tags（是否有 cluster tag）**

---

## 1. 算外連數（Outbound）

目的：知道每個節點總共「散出」多少連結，因為 PageRank 計算時需要用這個數字來分配權重。

- 把節點手動畫的所有出邊的 `link count` 加總
- 如果這個節點**不是 global 節點**，還要加上「所有 global 節點的 placements link count 之和」——因為每個普通頁面都被視為隱性連到全站導覽（Header、Footer 等）

---

## 2. 計算 PageRank 分數

PageRank 的核心概念：**一個頁面越多重要的頁面連向它，它就越重要。**

本工具採用 **Personalized PageRank**（Page & Brin 1998, §3 Personalization），針對站內分析的場景將 teleport 機率集中在首頁，而非原始論文預設的均勻分布。

### 初始狀態

每個節點的初始分數都是 1.0，接著反覆更新，直到所有節點的分數都不再有明顯變化為止（最多跑 100 輪，或分數變化量小於 0.0001 就提前停止）。

### 每輪更新邏輯

每個節點的新分數來自兩部分：

**① 基底分（Personalized Teleport — 偏向首頁）**

想像一個使用者在網站隨機亂點，有 15% 的機率他會「teleport」跳出當前連結路徑。在真實世界的站內情境，這相當於使用者**直接輸入網址**或**從 Google 搜尋結果點進來** —— 幾乎都是落到**首頁**。

因此所有 teleport 的質量（`N × 15%`，N 為節點數）都分配給 root node，其他節點基底分為 0。這讓首頁天然具有較高分數，反映它作為站內唯一入口的結構重要性，而不需要任何人工修正。

> 若沒有設定 root（未指定 `isRoot`），會 fallback 成原始論文的均勻 teleport（每個節點分到 15%）。

**② 從其他節點傳來的分數**

剩下 85% 的流量沿著連結傳遞。一個節點 A 如果連向節點 B，它會把自己部分分數「傳」給 B。傳多少取決於：
- A 的當前分數
- 這條邊的 link count（每頁平均放幾條連結）× B 的 page count（B 代表幾個實體頁面）
- 除以 A 的總加權外連數（確保 A 的分數不會憑空增加）

### 同 Cluster 加成

如果連結兩端的節點**共享至少一個 cluster tag**，這條邊的 link count 會乘以 **1.5**。代表主題一致的內部連結對 SEO 效果更好，分數傳遞效率更高。

### 隱性 Global 邊

每個普通節點都被視為自動連向所有 global 節點（Header、Footer 等）。這些隱性連結也參與分數計算，不需要手動畫邊。

### Self-loop 排除

一個節點連到自己（例如 product 頁面的「相關商品」推薦，template 自己連自己）**不會**進入 PageRank 的計算。原因：
- Self-loop 是 UX 設計（「你可能也喜歡」），不是真實的 authority 傳遞
- 若納入計算，高 pageCount 節點的 self-loop 會把 rank 鎖死在自己身上（例如 `6 × 357,000 = 2,142,000` 會完全主導該節點的外連分母，導致 rank 無法流向其他節點）

### 沒有外連的節點（Dangling）

如果某節點完全沒有出邊，它的分數不會憑空消失，而是依據 teleport vector 分配——因此 dangling rank 也會流向 root（若有設定），模擬使用者到死路後重新回到首頁的行為。這維持「所有節點分數總和 = N」的守恆特性。

---

### 為什麼 bridge 頁面（pageCount 小）分數會偏低？

這是 pageCount 加權設計的**預期行為**，不是 bug。

舉例：假設有一個 list 頁（pageCount=12）連往 product 頁（pageCount=357,000）。任何第三方節點同時連往這兩個目標時：
- 分配給 product 的權重比例 ≈ `linkCount × 357,000`
- 分配給 list 的權重比例 ≈ `linkCount × 12`

即使 linkCount 相同，product 也會吸走近 100% 的 rank —— 因為它代表的 URL 數量多出四個數量級。

**語意上**：本工具的分數反映「這個 template 在全站 SEO 中分到的總 rank 量」，不是「單一 URL 的競爭力」。覆蓋 357,000 頁的 product template 拿到比 12 頁的 list template 多得多的 rank，這符合 crawl budget 的直覺。

若需要看「平均每個 URL 的分數」，可自行除以 pageCount 得到 per-page 視角。

## 3. Tier 分類（high / mid / low）

分層**不是看分數的絕對值**，而是看**在所有節點裡的相對排名**，把全部節點大致切成三等份：

- 前 1/3 → **high**（綠色）
- 中 1/3 → **mid**（黃色）
- 後 1/3 → **low**（紅色）
- 只有一個節點、或所有分數都一樣 → **neutral**（靛色）

分數完全相同的節點（Tie）處理方式：卡在 high/mid 邊界時偏向 high；卡在 mid/low 邊界時偏向 low。

---

## 4. Weak Node 偵測

Tier 是相對排名，Weak 是統計上的異常偵測，兩者概念不同。

計算方式：
1. 算出所有節點分數的**平均值**與**標準差**
2. 分數低於「平均 − 1 標準差」的節點標記為 Weak

如果所有節點分數幾乎一樣（標準差 < 0.00001），就不標記任何節點。

> 一個節點可以同時是「low tier」又不是 Weak（只是相對偏低但不異常），也可以是「mid tier」但卻是 Weak（整體分數都高，但這個特別低）。

---

## 5. 爬取深度（Crawl Depth）

模擬搜尋引擎爬蟲從首頁（root）出發，沿著連結走，計算每個節點需要幾步才能到達。

- Root 本身 = 深度 0
- Root 直接連到的頁面 = 深度 1，以此類推
- 深度 **> 3** 會出現警告（搜尋引擎通常較不容易爬到深層頁面）
- 完全走不到的節點 = 深度 Infinity，顯示為 Unreachable

Global 節點的隱性連結也算在爬取路徑內，所以即使沒有手動畫邊，只要任何普通頁面可到達，global 節點就可達。

---

## 6. 孤島節點偵測（Orphan）

**沒有任何節點連向它**的頁面（Root 除外）。

包含隱性連結：Global 節點只要存在非 global 節點，就算有隱性入邊，不會被標為孤島。

孤島和 Weak 是兩個獨立問題：
- Weak = 有人連它，但連的不夠多或不夠重要，分數偏低
- Orphan = 根本沒有人連它，PageRank 無法傳進來

---

## 7. 健康狀態評估

每個節點有三個獨立的健康指標，任一個出問題都會顯示警告：

| 指標 | 警告條件 |
|------|---------|
| **Links** | 外連數 > 150（連結太多會稀釋每條連結的權重） |
| **Depth** | 爬取深度 > 3，或完全不可達 |
| **Tags** | 沒有設定任何 cluster tag |

如果沒有設定 root，Depth 指標顯示「N/A」而非警告。

---

## 小結

```
編輯 graph
  │
  ├─ 算外連數 → 每個節點放出多少連結
  ├─ 跑 Personalized PageRank → teleport 偏向首頁，self-loop 排除
  ├─ 分 Tier → high / mid / low（相對排名）
  ├─ 找 Weak → 統計上異常偏低的節點
  ├─ 算爬取深度 → 從首頁幾步可到
  ├─ 找孤島 → 完全沒人連的節點
  └─ 健康評估 → 外連數 / 深度 / tags 三項指標
```

所有步驟在每次 graph 變動後即時重跑。
